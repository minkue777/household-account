# Refactor (엄격한 리팩토링)

당신은 엄격한 리팩토링 전문가입니다. 아래 원칙을 **예외 없이** 적용합니다.

## 입력

사용자가 리팩토링 대상을 지정합니다: $ARGUMENTS

- 파일 경로, 디렉토리, 또는 기능 영역이 올 수 있습니다
- 대상이 비어있으면 **프로젝트 전체 스캔** 모드로 동작합니다

### 전체 스캔 모드 (대상 미지정 시)

1. `web/src/`, `functions/src/`, `android/` 하위의 모든 소스 파일을 탐색한다
2. 3대 원칙 위반을 **파일 간 중복** 중심으로 찾는다
3. 발견된 위반을 심각도순으로 정렬하여 보고한다
4. 사용자가 수정할 항목을 선택하면 해당 항목만 리팩토링한다

전체 스캔 시 우선순위:
- P0: 동일 로직이 3곳 이상 중복
- P1: 동일 로직이 2곳 중복
- P2: 500줄 이상 파일
- P3: 파일명과 내용 불일치

## 3대 원칙

### 원칙 1: Single Source of Truth — 코드는 반드시 한번만

- 동일한 로직이 2곳 이상에 존재하면 반드시 하나로 통합한다
- "동일"의 기준: 의도(intent)가 같은 코드. 우연히 비슷한 코드는 통합하지 않는다
- 통합 시 새 추상화(함수/컴포넌트/Hook)의 이름이 역할을 정확히 설명해야 한다
- 이름을 짓기 어렵다면 잘못된 추상화일 가능성이 높다 — 통합하지 않는다

**위반 감지 체크리스트:**
- [ ] 동일한 Firestore 쿼리 패턴이 여러 파일에 존재하는가
- [ ] 같은 UI 패턴(모달 래퍼, 폼 필드 그룹 등)이 복사되어 있는가
- [ ] 같은 비즈니스 규칙이 여러 곳에 하드코딩되어 있는가
- [ ] 상수/매직 넘버가 여러 곳에 흩어져 있는가

### 원칙 2: Discoverability — 파일명만으로 위치를 예측 가능

- 코드를 처음 보는 사람이 "이 기능은 어디 있을까?"라고 생각했을 때 정확히 그 위치에 있어야 한다
- 파일 하나는 하나의 명확한 주제를 가진다
- 파일명은 내용물을 정확히 설명한다 (e.g. `settlementService.ts`는 정산 로직만)
- 디렉토리 구조는 도메인을 반영한다

**위반 감지 체크리스트:**
- [ ] 한 파일에 관련 없는 함수들이 섞여 있는가
- [ ] 파일명과 실제 내용이 일치하지 않는가
- [ ] 유틸 함수가 비대한 "잡동사니" 파일에 들어있는가
- [ ] 찾으려는 코드의 위치가 직관적이지 않은가

### 원칙 3: Single Responsibility — 하나의 단위, 하나의 책임

- **함수**: 하나의 동작만 수행. "그리고(and)"로 설명해야 하면 분리 대상
- **컴포넌트**: 하나의 UI 관심사만 담당
- **Hook**: 하나의 상태 도메인만 관리
- **파일**: 하나의 모듈 역할만 수행

**위반 감지 체크리스트:**
- [ ] 함수가 데이터 가공과 사이드 이펙트를 동시에 하는가
- [ ] 컴포넌트가 200줄을 넘으면서 분리 가능한 UI 블록이 보이는가
- [ ] Hook이 서로 관련 없는 상태를 함께 관리하는가

## 보조 원칙

### 보조 원칙 1: Dependency Direction — 의존성은 한 방향으로만

- UI → 비즈니스 로직 → 타입/모델 순서로만 참조한다
- 상위 레이어(로직, 서비스)가 하위 레이어(UI, 컴포넌트)를 import하면 위반이다
- 순환 참조(A → B → A)는 반드시 제거한다
- 어떤 레이어가 어떤 디렉토리에 있는지는 프로젝트 구조에 따라 판단한다

**위반 감지 체크리스트:**
- [ ] 서비스/유틸 파일이 UI 컴포넌트를 import하고 있는가
- [ ] 두 모듈이 서로를 import하는 순환 참조가 있는가
- [ ] 타입 정의 파일이 런타임 로직에 의존하고 있는가

### 보조 원칙 2: Colocation — 관련 코드는 가까이

- 단일 소비자만 있는 코드는 소비자와 같은 위치(또는 가장 가까운 곳)에 둔다
- 여러 소비자가 생기는 시점에 공통 위치로 승격한다
- "혹시 나중에 쓸까봐" 미리 공통 위치에 두지 않는다

### 보조 원칙 3: Command-Query Separation — 변경과 조회를 섞지 않는다

- 데이터를 변경하는 함수(Command)와 조회하는 함수(Query)를 분리한다
- 하나의 함수가 "상태를 바꾸면서 동시에 값을 반환"하면 분리 대상이다
- 예외: 낙관적 업데이트 등 UX상 불가피한 경우

## 절대 하지 않는 것 (Anti-patterns)

1. **한 곳에서만 쓰이는 코드를 추출하지 않는다** — 재사용이 아니면 추상화 금지
2. **미래를 위한 설계를 하지 않는다** — 현재 필요한 최소한의 구조만
3. **prop이 6개 이상인 공통 컴포넌트를 만들지 않는다** — 과도한 설정은 가독성을 해침
4. **이름 짓기 어려운 추상화를 만들지 않는다** — 명확한 이름이 없으면 잘못된 분리
5. **파일 분리의 기준은 줄 수가 아니라 의도의 명확성이다** — 3줄이라도 독립된 의도가 있으면 분리, 단순히 파일을 나누기 위한 분리는 금지
6. **기존 동작을 변경하지 않는다** — 리팩토링은 구조만 바꾼다

## 실행 워크플로우

### Phase 1: 분석

1. 대상 코드를 전부 읽는다
2. 3대 원칙 각각의 위반 사항을 목록화한다
3. 위반이 없으면 "리팩토링 불필요"라고 보고하고 종료한다

### Phase 2: 계획 (반드시 사용자 승인 필요)

위반 사항별로 다음을 제시한다:

```
## 리팩토링 계획

### [원칙 위반 유형] 위반 #N: [한줄 설명]
- 현재: [어디서 어떻게 위반되고 있는지]
- 변경: [구체적으로 무엇을 어떻게 바꿀지]
- 영향 파일: [변경되는 파일 목록]
```

**이 계획을 사용자가 승인할 때까지 코드를 수정하지 않는다.**

### Phase 3: 실행

1. 테스트가 있으면 먼저 실행하여 현재 상태 확인
2. 계획대로 코드 수정
3. 빌드 확인 (`npm run build`)
4. 테스트가 있으면 재실행하여 통과 확인
5. 결과 보고

### Phase 4: 검증 보고

```
## 리팩토링 결과

- 변경 파일: N개
- 삭제 파일: N개
- 신규 파일: N개
- 순 코드 변화: +N / -N 줄
- 빌드: 통과/실패
- 테스트: 통과/실패/없음
```

## 판단 기준표

| 상황 | 판단 |
|------|------|
| 같은 코드가 2곳 | 통합 검토 (맥락에 따라) |
| 같은 코드가 3곳 이상 | 반드시 통합 |
| 파일이 300줄 이상 | 분리 검토 |
| 파일이 500줄 이상 | 반드시 분리 |
| 함수가 30줄 이상 | 분리 검토 |
| 함수가 50줄 이상 | 반드시 분리 |
| 공통 컴포넌트 props > 5 | 설계 재검토 |
| Hook이 3개 이상 상태 관리 | 분리 검토 |

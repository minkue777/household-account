# Refactor (엄격한 리팩토링)

이 스킬은 구조 개선과 중복 제거를 수행할 때 사용한다.  
목표는 동작을 바꾸지 않고 코드 품질을 높이는 것이다.

## 입력

사용자가 리팩토링 대상을 지정한다: `$ARGUMENTS`

- 파일 경로, 디렉터리, 기능 영역을 받을 수 있다.
- 대상이 비어 있으면 **프로젝트 전체 스캔 모드**로 동작한다.

## 전체 스캔 모드 (대상 미지정 시)

1. `web/src/`, `functions/src/`, `android/` 하위의 모든 소스 파일을 탐색한다.
2. 중복 로직, 책임 혼재, 탐색성 저하 지점을 찾는다.
3. 발견 항목을 심각도(P0~P2) 순으로 정렬해 제시한다.
4. 사용자 확인 후 선택된 항목부터 순차적으로 수정한다.

### 스캔 제외 경로

- 빌드/생성물: `web/.next/`, `web/out/`, `functions/lib/`, `android/**/build/`
- 의존성/캐시: `**/node_modules/`, `**/.gradle/`
- 생성 소스맵/번들: `**/*.map`, `**/*.min.js`

## 핵심 원칙

### 1) SSOT (Single Source of Truth)

- 같은 의도의 로직이 2곳 이상이면 공통화 후보로 본다.
- 공통화 단위는 함수, 훅, 컴포넌트, 유틸 중 가장 좁은 범위를 선택한다.
- 메시지 문구/검증 규칙/매직 넘버도 SSOT 대상으로 본다.

### 2) Discoverability (찾기 쉬운 구조)

- 파일명만 보고 책임을 예측할 수 있어야 한다.
- 한 파일에는 한 주제를 유지한다.
- 공통 로직은 도메인 기준 위치(`lib/utils`, `lib/*Service`, `components/common`)에 둔다.

### 3) SRP (Single Responsibility Principle)

- 함수는 한 가지 목적만 수행한다.
- 컴포넌트는 UI 조합, 훅은 상태/동작, 유틸은 순수 계산에 집중한다.
- 데이터 변경(Command)과 조회(Query)를 가능한 분리한다.

### 4) 동작 보존 (Behavior Lock)

- 리팩토링은 구조 변경이 목적이며 기능 변경이 아니다.
- 기존 UX 문구, 경고 조건, API 계약, 저장 스키마를 임의로 바꾸지 않는다.
- 변경이 불가피하면 사용자에게 이유와 영향을 먼저 알린다.

### 5) 인코딩/문자열 안전성

- UTF-8 인코딩을 유지한다.
- 대규모 치환/재저장으로 한글 깨짐이 발생하지 않게 한다.
- 깨짐 징후가 보이면 즉시 중단하고 안정 경로(국소 수정)로 전환한다.

### 6) Diff 최소화

- 의미 없는 대량 재포맷/전면 재작성은 피한다.
- 필요한 블록만 국소 수정한다.
- 리뷰 가능한 크기의 커밋 단위로 쪼갠다.

## 프로젝트 특화 권장 원칙

1. 도메인 규칙 상수화
- 월별 분할 조건, 에러 문구, 검증 메시지는 상수/유틸로 중앙화한다.

2. 모달/폼 패턴 재사용
- Add/Edit 쌍은 `필드 UI`, `폼 상태 훅`, `검증/패치 유틸`로 분해해 재사용한다.

3. 서비스 계층 일원화
- Firestore/외부 API 호출은 서비스 계층으로 모으고 UI에서 직접 중복 호출하지 않는다.

4. 플랫폼 분기 캡슐화
- Android/Web 분기 조건은 브리지/플랫폼 유틸에 모아 컴포넌트 분기를 줄인다.

5. 생성물 리팩토링 금지
- `functions/lib` 같은 빌드 산출물은 수정 대상에서 제외한다.

## 기존 규칙에서 삭제/완화한 항목

- 파일 줄 수(300/500) 같은 고정 임계치는 삭제한다.
- `props > N`, `hook state > N` 같은 숫자 규칙은 삭제한다.
- 절대값 임계치 대신 **의도/복잡도/변경 위험**으로 판단한다.
- "무조건 계획 확인 후 수정"은 완화한다.
- 소규모 안전 변경은 바로 수행하고, 대규모/고위험 변경만 사전 확인한다.

## 실행 워크플로

### Phase 1: 분석

1. 대상 범위 파일을 수집한다 (`rg --files` 우선).
2. 중복/책임혼재/탐색성 이슈를 목록화한다.
3. 심각도와 수정 난이도를 함께 기록한다.

### Phase 2: 계획

- 수정 전 다음 형식으로 계획을 제시한다.

```md
## 리팩토링 계획

### [원칙] 항목 제목
- 현재: 문제 지점
- 변경: 수정 방식
- 영향 파일: 파일 목록
- 리스크: 회귀 가능성/검증 방법
```

### Phase 3: 구현

1. 공통 단위(유틸/훅/컴포넌트)를 먼저 만든다.
2. 호출부를 점진적으로 치환한다.
3. 동작 변경이 없는지 중간 검증한다.

### Phase 4: 검증

- 변경 범위별로 최소 검증을 수행한다.
- `web` 변경: `cd web && npm run build`
- `functions` 변경: `cd functions && npm run build`
- `android` 변경: 필요 시 `cd android && ./gradlew assembleDebug` (Windows: `gradlew.bat`)

### Phase 5: 결과 보고

```md
## 리팩토링 결과

- 변경 파일: N개
- 추가 파일: N개
- 삭제 파일: N개
- 변경 요약: 핵심 구조 변경 사항
- 검증: 실행한 명령과 결과
- 잔여 리스크: 남은 중복/추가 개선 후보
```

## 금지 사항

1. 산출물/캐시 폴더를 리팩토링하지 않는다.
2. 기능 변경을 리팩토링으로 포장하지 않는다.
3. 이름이 모호한 추상화를 만들지 않는다.
4. 관련 없는 파일까지 함께 건드리지 않는다.
